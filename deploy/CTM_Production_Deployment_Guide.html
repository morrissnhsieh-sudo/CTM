DEPLOYMENT GUIDE

Deploying CTM to Production

Step-by-step instructions to deploy the full CTM Collaborative AI Spreadsheet Platform on a Linux server with Docker, Nginx, and Let's Encrypt SSL.

⏱ Estimated time: 30–60 minutes 🐧 Ubuntu 22.04 / Debian 12 🐳 Docker Compose 📅 May 2026

📋 What You Need Before Starting

RequirementMinimumRecommendedNotes
Server OS Ubuntu 22.04 or Debian 12 Ubuntu 22.04 LTS Any VPS or cloud VM (AWS, GCP, Azure, Hetzner, DigitalOcean)
CPU 2 vCPU 8 vCPU Kafka + Keycloak + AI service are CPU-intensive
RAM 8 GB 32 GB Keycloak needs ~1 GB, ai-service needs ~2–4 GB
Disk 40 GB SSD 200 GB SSD For Docker images, PostgreSQL data, and file uploads
Domain Any registered domain with DNS control e.g. ctm.example.com
Ports open 22 (SSH), 80 (HTTP), 443 (HTTPS) All other ports stay internal — only Nginx is public
Vertex AI key Service account JSON from Google Cloud Already at VertexKeys/d-sxd110x-ssd1-aaos-34f80b5f4448.json
💡
Cloud provider recommendations Hetzner CCX33 (8 vCPU, 32 GB, ~€40/mo) offers the best value. AWS t3.2xlarge or GCP n2-standard-8 also work well. For testing, a 4 GB RAM VPS is sufficient but expect slower AI responses.

🏗️ Production Architecture

In production, all traffic enters through Nginx, which handles SSL termination and routes to the appropriate service. All services run in a private Docker network and are never directly exposed to the internet.

Internet │ HTTP :80 ──→ redirect to HTTPS │ HTTPS :443 ▼ ┌─ Nginx (SSL termination) ─────────────────────────────────────────────┐ │ app.domain ──→ frontend:3000 (Next.js UI) │ api.domain ──→ api-service:3001 (REST API + MCP) │ auth.domain ──→ keycloak:8080 (OAuth 2.0 / OIDC) │ collab.domain ──→ collab-service:1234 (WebSocket CRDT) │ chat.domain ──→ messaging-service:3002 (Socket.io) └───────────────────────────────────────────────────────────────────────┘ │ (internal ctm-net Docker network — no external access) ▼ Application Services Infrastructure api-service (Node.js) postgres:5432 (PostgreSQL 16 + pgvector) collab-service (Node.js) redis:6379 (Redis 7.2 cache + pub/sub) pm-service (Go) kafka:9092 (Kafka 3.7 event bus) ai-service (Python) minio:9000 (S3-compatible storage) messaging-svc (Node.js) keycloak:8080 (Auth / SSO) frontend (Next.js)

🔄 Development vs Production

The production compose file (deploy/docker-compose.prod.yml) differs from the development one in several important ways:

AreaDevelopmentProduction
Passwords ctm_dev_pass (hardcoded) Docker Secrets from /etc/ctm/secrets/
Keycloak mode start-dev (no TLS required) start (production, proxy=edge)
SSL / TLS None Let's Encrypt via Certbot + Nginx
Nginx Not included Rate limiting, security headers, WebSocket upgrade
Restart policy None (manual restart) restart: always on all services
External ports All services exposed (3001, 5432, 6379…) Only 80 + 443 via Nginx
PostgreSQL buffers 256 MB shared_buffers 8 GB shared_buffers (32 GB server)
AI service workers 1 uvicorn worker 4 uvicorn workers
Certificate renewal — Certbot container auto-renews every 12 hours

🚀 Deployment Steps

1
Provision a Server
Create a Linux VM with the right specs

Create a new server with any cloud provider. Ubuntu 22.04 LTS is recommended.

ProviderRecommended PlanApprox. Cost
🟠 HetznerCCX33 (8 vCPU, 32 GB)~€40/month
🔵 DigitalOceanGeneral Purpose 8 vCPU 32 GB~$160/month
🟡 AWSt3.2xlarge (8 vCPU, 32 GB)~$270/month
🔴 GCPn2-standard-8 (8 vCPU, 32 GB)~$310/month
🟣 VultrCloud Compute 8 vCPU 32 GB~$160/month
⚠️
Minimum for testing: 4 GB RAM You can run CTM on 4 GB RAM but Kafka and Keycloak may be slow. For team use or production load, 16–32 GB is strongly recommended.

After your server is created, note its public IP address and SSH in:

terminal
ssh root@YOUR_SERVER_IP
2
Configure DNS
Point your domain and subdomains at the server

In your domain registrar's DNS panel, add these A records pointing to your server's IP address:

Record TypeNameValuePurpose
Actm.example.comYOUR_IPRoot domain
Aapp.ctm.example.comYOUR_IPFrontend UI
Aapi.ctm.example.comYOUR_IPREST API
Aauth.ctm.example.comYOUR_IPKeycloak SSO
Acollab.ctm.example.comYOUR_IPWebSocket CRDT
Achat.ctm.example.comYOUR_IPSocket.io chat
💡
DNS propagation takes up to 48 hours You can proceed with the remaining steps while DNS propagates. Verify propagation with: dig app.ctm.example.com or dnschecker.org
3
Set Up the Server
Install Docker, configure firewall, tune kernel

Run the automated setup script as root on your server. This installs Docker, configures UFW (only ports 22, 80, 443 open), and tunes kernel parameters for production workloads.

server — bash
# On your server (SSH'd in as root):
curl -fsSL https://raw.githubusercontent.com/morrissnhsieh-sudo/CTM/main/deploy/scripts/setup-server.sh | bash

Or if you've already cloned the repo:

server — bash
bash deploy/scripts/setup-server.sh

The script performs these actions:

  • Updates system packages (apt-get upgrade)
  • Installs Docker Engine (latest stable) and enables it as a service
  • Configures UFW firewall: allow SSH (22), HTTP (80), HTTPS (443); deny everything else
  • Installs Certbot for SSL certificate management
  • Creates /etc/ctm/secrets/ with mode 700
  • Tunes kernel: net.core.somaxconn=65535, fs.file-max=1000000
  • Increases open file limits for high-concurrency workloads
⚠️
Log out and back in after setup The script adds your user to the docker group. This change only takes effect after a new SSH session. Run exit and SSH back in before proceeding.
4
Clone the Repository
Get the CTM source code on the server
server — bash
# Clone the repository
git clone https://github.com/morrissnhsieh-sudo/CTM.git
cd CTM

The deploy/ directory contains all production configuration files:

directory structure
deploy/
├── docker-compose.prod.yml   # Production Compose file
├── nginx/
│   ├── nginx.conf            # Nginx main config
│   └── conf.d/ctm.conf       # Virtual host config (SSL, routing)
└── scripts/
    ├── setup-server.sh       # Server setup (Step 3)
    ├── setup-secrets.sh      # Generate secrets (Step 5)
    ├── get-ssl-cert.sh       # Obtain SSL certs (Step 7)
    └── deploy.sh             # Full deployment (Step 8)
5
Generate Secrets
Create cryptographically secure passwords for all services

The production deployment uses Docker Secrets — passwords are stored as files on the host filesystem, never as plaintext environment variables. The script generates strong random passwords for all services.

server — bash
sudo bash deploy/scripts/setup-secrets.sh

This creates the following files in /etc/ctm/secrets/ with mode 600 (owner-only read):

FileUsed byContents
pg_password.txtPostgreSQL, Keycloak32-char random password
redis_password.txtRedis32-char random password
minio_password.txtMinIO S332-char random password
keycloak_admin_password.txtKeycloak admin console32-char random password
vertex_key.jsonM6 AI ServiceGoogle Cloud service account JSON

Next, copy the Vertex AI key from your local machine to the server:

your local machine — PowerShell
# Run this on YOUR local machine, not the server
scp "C:\Users\User\Code\VertexKeys\d-sxd110x-ssd1-aaos-34f80b5f4448.json" `
    root@YOUR_SERVER_IP:/etc/ctm/secrets/vertex_key.json
🔐
Save the generated passwords immediately The script displays the Keycloak admin and PostgreSQL passwords once. Copy them to a secure password manager (1Password, Bitwarden, etc.) before closing the terminal. They cannot be recovered without resetting, which requires restarting affected services.
6
Configure Production Environment
Create .env.prod with your domain and API keys
server — bash
cp .env.prod.example .env.prod
nano .env.prod

The minimum required changes are:

.env.prod
# Replace with your actual domain
DOMAIN=ctm.example.com

# Generate two random secrets:
#   openssl rand -base64 32
INTERNAL_TOKEN_SECRET=paste-32-char-random-string-here
NEXTAUTH_SECRET=paste-another-32-char-random-string-here

# Vertex AI project (already configured)
VERTEX_PROJECT_ID=d-sxd110x-ssd1-aaos
VERTEX_REGION=us-east5

# Optional: email notifications
RESEND_API_KEY=re_your_resend_key

Generate the two random secrets with this command:

server — bash
openssl rand -base64 32
💡
Database passwords come from secrets files, not .env.prod You do not need to set PG_PASSWORD or REDIS_PASSWORD in .env.prod — they are read automatically from /etc/ctm/secrets/*.txt by Docker Secrets.
7
Obtain SSL Certificates
Free Let's Encrypt certificates via Certbot

First, start Nginx on port 80 so Certbot can complete the ACME HTTP-01 challenge:

server — bash
# Start Nginx (HTTP only — SSL not configured yet)
docker compose -f deploy/docker-compose.prod.yml up -d nginx

Then obtain certificates for all subdomains:

server — bash
bash deploy/scripts/get-ssl-cert.sh ctm.example.com admin@example.com

This requests a wildcard-style certificate covering:

  • ctm.example.com
  • app.ctm.example.com
  • api.ctm.example.com
  • auth.ctm.example.com
  • collab.ctm.example.com
  • chat.ctm.example.com

The script also adds a daily cron job for automatic renewal:

cron (auto-added)
0 12 * * * certbot renew --quiet
⚠️
DNS must be propagated before this step Certbot contacts Let's Encrypt servers which resolve your domain to verify ownership. If DNS hasn't propagated yet, the certificate request will fail. Wait for dig app.ctm.example.com to return your server's IP.

After getting certificates, update the Nginx config with your actual domain:

server — bash
sed -i 's/ctm\.example\.com/ctm.yourdomain.com/g' deploy/nginx/conf.d/ctm.conf
8
Deploy CTM
Build all images and start all 12 services
server — bash
bash deploy/scripts/deploy.sh

The deployment script does the following, automatically:

StageWhat happensDuration
1. Pre-flightChecks Docker, secrets, .env.prod~5 sec
2. ImagesBuilds all 6 service images from source5–15 min (first time)
3. InfrastructureStarts PostgreSQL, Redis, Kafka, MinIO; waits for health1–2 min
4. KeycloakStarts auth server; waits for /health/ready60–90 sec
5. App servicesStarts M2–M7 microservices~30 sec
6. FrontendStarts Next.js production server~20 sec
7. CertbotStarts auto-renewal daemon~5 sec

To force a full rebuild of all images (e.g. after a code update):

server — bash
bash deploy/scripts/deploy.sh --rebuild

First login

The default user accounts from the Keycloak realm export are not suitable for production. Create a proper admin account:

  1. Open https://auth.ctm.example.com/admin
  2. Log in with Keycloak admin credentials (from /etc/ctm/secrets/keycloak_admin_password.txt)
  3. Navigate to CTM realm → Users → Add user
  4. Create your production admin user, set a strong password, assign OWNER role
  5. Delete the demo accounts: demo@ctm.app and admin@ctm.app

🔄 Updating CTM

To deploy a new version after code changes:

server — bash
# Pull latest code
git pull origin main

# Rebuild and redeploy
bash deploy/scripts/deploy.sh --rebuild

The deployment is nearly zero-downtime: Compose starts new containers before stopping old ones when restart: always is set. The typical update takes 5–10 minutes.

💡
Database migrations run automatically SQL migration files in infra/postgres/migrations/ are applied via the docker-entrypoint-initdb.d volume mount. New migrations are applied on the next container start. Existing data is never dropped.

💾 Backups

PostgreSQL backup (daily)

Add this cron job on the server to create daily database dumps:

server — bash (add to crontab: crontab -e)
# Daily PostgreSQL backup at 2:00 AM
0 2 * * * docker exec ctm-postgres pg_dump -U ctm ctm | gzip > /backups/ctm-$(date +\%Y\%m\%d).sql.gz

# Keep only last 30 days
0 3 * * * find /backups -name "ctm-*.sql.gz" -mtime +30 -delete

Create the backup directory first:

server — bash
mkdir -p /backups && chmod 700 /backups

Restore from backup

server — bash
# Stop the application services (keep postgres running)
docker compose -f deploy/docker-compose.prod.yml stop api-service collab-service

# Restore
gunzip -c /backups/ctm-20260101.sql.gz | docker exec -i ctm-postgres psql -U ctm ctm

# Restart services
docker compose -f deploy/docker-compose.prod.yml start api-service collab-service

MinIO (file uploads) backup

server — bash
# Sync MinIO buckets to a local directory
docker run --rm --network ctm_ctm-net \
  -e MINIO_USER=$(cat /etc/ctm/secrets/minio_user.txt) \
  -e MINIO_PASS=$(cat /etc/ctm/secrets/minio_password.txt) \
  minio/mc sh -c \
  "mc alias set local http://minio:9000 \$MINIO_USER \$MINIO_PASS && \
   mc mirror local/ctm-attachments /backup/attachments"

📈 Scaling

To handle more users, you can scale the stateless application services horizontally:

server — bash
# Scale API Gateway to 3 replicas
docker compose -f deploy/docker-compose.prod.yml up -d --scale api-service=3

# Scale AI service to 2 replicas
docker compose -f deploy/docker-compose.prod.yml up -d --scale ai-service=2
⚠️
Services that cannot be scaled on a single server
  • collab-service — uses Redis adapter for multi-pod scaling; OK to scale
  • postgres — single primary; use managed PostgreSQL (RDS, Cloud SQL) for HA
  • keycloak — needs Infinispan clustering for multi-pod; use managed identity for large scale
  • kafka — needs multi-broker KRaft quorum for HA; single broker is fine for <10k users

🔧 Troubleshooting

A container keeps restarting

server — bash
# View the last error
docker logs --tail=50 ctm-api

# Check exit code
docker inspect ctm-api --format "{{.State.ExitCode}}"

Keycloak cannot connect to PostgreSQL

This usually means PostgreSQL is not listening on the Docker network. Check:

server — bash
# Verify PostgreSQL is listening on all interfaces
docker exec ctm-postgres psql -U ctm -c "SHOW listen_addresses;"
# Expected: *

# If it shows 'localhost', restart PostgreSQL
docker compose -f deploy/docker-compose.prod.yml restart postgres

SSL certificate errors

server — bash
# Check certificate expiry
docker exec ctm-nginx openssl s_client -connect localhost:443 2>/dev/null | \
  openssl x509 -noout -dates

# Force certificate renewal
docker compose -f deploy/docker-compose.prod.yml run --rm certbot certbot renew --force-renewal

Out of disk space

server — bash
# See what's using space
docker system df -v

# Remove unused images and containers
docker system prune -f

# Remove old backup files
find /backups -name "*.sql.gz" -mtime +30 -delete

Reset the entire platform (WARNING: deletes all data)

server — bash
# Stop and remove all containers, volumes, and images
docker compose -f deploy/docker-compose.prod.yml down -v --rmi all

# Redeploy from scratch
bash deploy/scripts/deploy.sh --rebuild
🚨
The -v flag deletes all data volumes permanently This includes all spreadsheet data, user accounts, chat messages, and uploaded files. Always create a backup before running this command.
